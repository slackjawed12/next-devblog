### 제품을 생각하기

&nbsp; &nbsp; 재작년에 퇴사한 선임이 '아이디어 불패의 법칙'이라는 책을 추천해서, 여름에 읽었다.
올 해 가장 인상깊게 읽은 책이었다. 이 책을 조금이나마 일찍 접했으면 삶이 좀 바뀌었을까 싶기도 했다.
(물론 그러진 않았겠지만) 책을 다 읽고 들떠서, 동료들한테 이 책에 나온 내용을 적용해보자고 난리법석을 떨었다.
동료들이 호응해준 덕분에, 임시 제품팀을 4명으로 꾸려서 9월 한 달 동안 책에 나온 아이디어 검증 방법론을 적용했다.<br/>
&nbsp; &nbsp; 우리팀은 이 프로세스를 XYZ라고 줄여서 말했는데, 책의 시장호응가설의 표현방법인 **XYZ 가설**에서 따왔다.
XYZ 가설은 **'적어도 X 퍼센트의 Y(타겟 고객)는 Z할 것이다.'** 라는 방식으로 출시할 제품의 시장호응가설을 표현하는 방법인데,
한 문장 안에 우리 목표가 무엇인지 수치까지 포함해서 표현할 수 있어 의사소통이 편했다.<br/>
&nbsp; &nbsp; 이렇게 막상 들뜨긴 했는데, 어떻게 시작해야할 지 몰라서 일단 회의실에 모두 모여 각자 아이디어를 제시하기 시작했다.

![xyz](/images/posts/retro/xyz.png)
*노션에 작성한 xyz 미팅 기록*

&nbsp; &nbsp; 프리토타이핑 할 때는 정말 단순하게 기능을 구현하거나, 아예 기능 구현 없이 배너만 올릴 때도 있었다.
유저가 배너를 얼마나 클릭했는지에 따라서 아이디어를 버리거나 계속 진행했던 것이었다.
참고로 '오른다 vs 내린다 투표' 기능은 배포할 때마다 유저 반응이 가설 수치를 뛰어 넘었고, 
라이브 서비스에서 '퀴즈' 기능으로까지 발전해 유저들에게 제공 중이다.<br/>
&nbsp; &nbsp; 이 경험을 통해 스타트업에서 일하는 방식을 다시 생각하게 됐다.
이전에는 보통 위에서 요구시항 리스트로 제시하는 방식으로 제품을 출시했는데,
이 프로세스는 구성원이 임팩트에만 집중해서 제품을 출시하기 때문에 실행 속도도 빠르고, 결과도 좋았다.
물론 이게 정답이라는 건 아니지만, 효율적이면서 동시에 모든 구성원이 만족하는 제품 프로세스였다.<br/>
&nbsp; &nbsp; 이후 10월즈음에는 쇼핑몰 구매 시 리워드를 지급하는 기능을 테스트했는데,
생각보다 유저 반응이 없어서 빠르게 접을 수 있었다. 이 때도 쇼핑몰 구매 추적이나 리워드 지급은 전혀 구현하지 않았다.
그냥 유저의 배너 클릭 횟수만 추적했고, 예상치에 미치는지만 확인했다.<br/>
&nbsp; &nbsp; 이 방식에는 한계도 있었는데, '왜' 사람들이 이 기능을 쓰는지/안 쓰는지 알 수 없다는 것이었다.
우리는 XYZ 가설을 세울 때 데이터와 타겟 고객에만 집중했다. 
그러다보니 통했던 아이디어인 퀴즈 기능을 출시하고 나서, 
'어떤 기능을 추가로 출시해야하지?'라는 질문에 대답을 할 수 없었다.
아이디어가 나온 이후 빠르게 실행하는 방법은 이제 알겠는데, 아이디어 자체가 고갈되면 별 수가 없었다.<br/>

### 만든 것 - 블로그

&nbsp; &nbsp; 5월에는 nextjs로 직접 블로그를 만들어 배포했다. 최근까지 다크모드 만든다고 삽질했다.
원래 jekyll + github 블로그 배포로 운영했는데, 언젠가부터 로컬에서 ruby 실행이 잘 안되고 있었다.
디버깅하긴 귀찮고.. 글로벌 웹사이트 출시때문에 next 공부도 필요한 시점이어서 next14로 블로그를 만들었다.
나름 이렇게 만들고나니 뿌듯하긴 한데, 너무 뭐가 없어서 미완성처럼 보인다.
구글 광고도 좀 넣고, toc나 이전글, 다음글 같은 기능도 추가해야겠다.<br/>
&nbsp; &nbsp; 블로그 글은 일주일에 한 개 반드시 쓰자고 다짐했는데, 전혀 실천하고 있지 않다..
to-do list 앱인 things에 블로그 글로 쓸 소재만 잔뜩 쌓아두고 있다.
글을 쓸 때마다 몇 시간씩 소비하다보니, 괴롭고 오래걸리는 일 하기 싫어하듯 몸이 조건반사하는 것 같기도 하다.
새해에는 꼭 일주일에 한 개.. 지켜야겠다.

### 만든 것2 - 이력서 with next

&nbsp; &nbsp; '코딩인터뷰 완전분석' 책을 읽으면서 이력서 관련 파트가 인상깊어 next로 이력서 페이지를 만들었다.
분량이나 내용 측면에서, 책이 강조하는 부분과 내 이력서가 다른 점이 많아 아예 갈아엎어야겠다 싶었다.
기존 이력서는 Notion으로 업데이트 생길때마다 고쳤는데, 이게 생각보다 히스토리 관리가 힘들었다. 
Notion에서 7일보다 오래된 변경이력을 보려면 요금제 업그레이드가 필요하기 때문에 페이지 복붙으로 처리했는데, 차이점을 한 눈에 보기가 힘들었다.
또, 내용 업데이트하고 pdf로 추출할때마다 매번 사이즈 70이니 69니 마이크로 컨트롤하면서 출력 페이지 맞추는것도 고역이었다.
이걸 안하면 1줄씩 엇나가는 내용들이 있어서 안 할 수도 없는 노릇이었다.

&nbsp; &nbsp;그래서 Next로 이력서 내용을 옮겨서 Notion 이력서의 불편함을 좀 없애려고 했다.
아래 세 가지가 편해졌다.

1. 내용 변경이 곧 코드 변경이므로 이력서 히스토리가 git으로 관리됨
2. 컴포넌트화된 스타일
3. 템플릿 바꾸고 싶으면 AI 도움 받아서 바로 변경 가능

3번이 아주 편했는데, 무료 템플릿 이미지 gpt에게 던져주고 next 코드로 뽑아달라고 하니 바로 줬다.
그리고 이력서 내용들을 임팩트있게 꾸며서 옮기는 중이다.

### 옮긴 것 - Notion에서 Obsidian으로

&nbsp; &nbsp; 2024 새해부터 계속 시도했던게 Notion을 Obsidian으로 옮기는 것이었다.
이전까지 메모를 계속 Notion에 쌓아두고 있었는데, 단축키 활용도 불편했고, 웹 기반이라는 것도 불편했다.
Obsidian은 로컬 기반이라 빨랐고, vim 키맵을 제공해서 바로 갈아타려고 했다.
협업+공유용 메모는 Notion, 개인 메모는 Obsidian으로 구분해서 사용할 참이었다.
Notion에 있는 모든 페이지를 Markdown으로 추출해서 Obsidian으로 옮기기까지 했다.
문제는 오래도록 Notion을 써오다 보니, 눈이 Notion UI에 익숙해져서 Obsidian의 메모가 잘 안읽힌다는 것이었다.
게다가 Obsidian은 마크다운기반이라 WYSIWYG이 아니었다. 동작방식이 예상과 달라서 좀 혼란스러웠다.
&nbsp; &nbsp; 그래도 익숙함의 문제라 생각하고 플러그인 이것저것 갖다 깔고 하다보니,
연말쯤 와서야 Obsidian이 편해졌다. 
아직도 새폴더 생성 단축키 없는것이나 테이블 생성, bullet point, 숫자 리스트 작성하는 건 불편하다.
그래도 계속 발전중인 프로그램이니 더 사용성이 좋아지지 않을까 기대하면서 쓰고있다.<br/>

### 기술 - AWS

&nbsp; &nbsp; 올해 초부터 여름까지, 자격증을 3개를 땄다. AWS 자격증 2개, 정보처리기사 자격증 하나.
AWS 자격증은 AWSKRUG이라는 슬랙 채널에서 진행하는 스터디 덕분에 딸 수 있었다. 
그 때 스터디 진행하시던 시니어 분께서 말하신게 '자격증 약발 6개월 정도 간다'는 것이었는데,
정말 정확하다. AWS SAA를 3월에 땄는데 지금은 거의 지식이 남아있지 않은 느낌이다.
물론 의미 없었냐하면 그건 아니었다. SAA 공부하면서 알게된 DMS를 실제로 적용해서 DB 마이그레이션도 해봤고,
월 비용 절감도 50% 가까이 했다. 그래도 나름 한푼한푼 중요한 상황에서 임팩트 냈다고 할 수 있겠다.<br/>

### 기술 - nestjs

&nbsp; &nbsp; 11월에는 express로 돌리던 서버에 nest.js를 도입했다. 
class 간 의존성을 직접 짜넣는 원시적인 짓을 더 이상 하지 않아도 되어서 정말 편하다.
생각하지 못한 단점으로는 빌드타임이 많이 늘었다는 것이다.
기존에 express만 돌리던 때에는 1)tsc로 타입체크 + dts 생성, 2)babel로 트랜스파일
이렇게 진행했는데, nest는 tsc로만 빌드한다(asset파일은 별도로 복사)
monorepo의 의존성 패키지들은 이전과 큰 차이 없이 대개 2-3초 내로 빌드가 끝났는데,
nestjs cli로 실행하는 app 패키지들이 빌드가 완료되는데 30초가 넘게 걸린다.
swc도 써보고 했는데, tsc냐 swc냐의 문제가 아니라 증분빌드가 잘 작동하지 않는다는 느낌이어서,
일단 최대한 tsconfig 건드려가면서 빌드를 수정하려고 하는 중이다.<br/>
&nbsp; &nbsp; 그리고 prisma ORM을 도입했다. 
타입도 엄격하게 잡아주고, migrate 툴도 간단하게 사용할 수 있어 장점이 많았다.
그런데 join 쿼리가 예상과 정말 다르게 생성되기도 했고,
timezone 관련 설정이 없어, 직접 메서드에 들어가는 파라미터마다 timezone offset을 맞춰주는 작업이 필요했다.
그리고 지금도 계속 활발하게 업데이트되는 라이브러리다보니, 아직 없는 기능도 많다.
특히 createMany에서 배열 반환 안되는거는... 좀 답답했다.
아무튼 sequelize의 active-record 방식이 불편해서 결국 옮기긴 했는데, prisma도 나사빠진 구석이 꽤 많다.
